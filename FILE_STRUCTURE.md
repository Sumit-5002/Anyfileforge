## 📁 `src/` — Frontend Source Code

This is the heart of the application. It follows a **feature-based architecture** where each page or domain has its own self-contained folder.

```
src/
├── 📄 main.jsx             # React app entry point — mounts <App>, sets up i18n & React Router
├── 📄 vite-env.d.ts        # TypeScript type declarations for Vite environment variables
│
├── 📁 app/                 # Root application shell
├── 📁 components/          # Reusable, shared UI components
├── 📁 config/              # App-wide configuration (Firebase, i18n)
├── 📁 contexts/            # React Context providers (global state)
├── 📁 data/                # Static data files (tool definitions, categories)
├── 📁 features/            # Page-level feature modules (one folder per route/page)
├── 📁 hooks/               # Custom React hooks (reusable stateful logic)
├── 📁 locales/             # i18n translation files (EN, ES, HI)
├── 📁 services/            # Business logic — file processing, Firebase API calls
├── 📁 styles/              # Global CSS styles and design tokens
├── 📁 tools/               # Specialized interactive tool implementations
└── 📁 utils/               # Pure utility/helper functions
```

### 📁 `src/app/` — Application Shell

```
app/
├── 📄 App.jsx          # Root component — defines all React Router routes and layout
├── 📄 App.css          # Root-level styles (minimal, mostly handled by index.css)

| File | Purpose |
|------|---------|
| `App.jsx` | Wraps the entire app in `<AuthProvider>`, `<BrowserRouter>`, and `<DnDProvider>`. Defines all page routes using React Router v7. |
| `ScrollToTop.jsx` | Listens to `useLocation()` and calls `window.scrollTo(0,0)` on navigation. |

### 📁 `src/contexts/` — React Context (Global State)

```
contexts/
└── 📄 AuthContext.jsx  # Authentication context — provides currentUser to all components
```

`AuthContext.jsx` wraps the app with a provider that listens to Firebase `onAuthStateChanged` and exposes `{ currentUser, loading }` via `useContext(AuthContext)`.

---

### 📁 `src/data/` — Static Data

```
data/
└── 📄 toolsData.js     # Master list of all tools — name, slug, category, icon, description
```

`toolsData.js` is the single source of truth for tool metadata. It exports an array of tool config objects that drive both the `ToolsPage` listing and the dynamic routing in `App.jsx`. Each tool entry maps a URL slug to its runner component.

## 📁 `server/` — Node.js / Express Backend

An **optional** backend server for processing tasks that are too heavy for the browser (e.g., large PDF compression, image operations using `sharp`).

```
server/
├── 📄 server.js            # Express app entry point — middleware, CORS, rate limiting, routes
├── 📄 package.json         # Server-specific dependencies (Express, Sharp, Multer, etc.)
├── 📄 .env                 # Server environment variables (PORT, allowed origins)
├── 📄 .gitignore           # Excludes node_modules and uploads from git
├── 📄 README.md            # Server-specific setup and API documentation
│
├── 📁 routes/              # Express route handlers
│   ├── 📄 pdf.js           # POST /api/pdf/* — PDF processing endpoints (compress, convert)
│   ├── 📄 image.js         # POST /api/image/* — Image processing endpoints (sharp-based)
│   ├── 📄 engineer.js      # POST /api/engineer/* — Developer tool backend endpoints
│   └── 📄 researcher.js    # POST /api/researcher/* — Researcher tool backend endpoints
│
└── 📁 uploads/             # Temporary file storage for incoming uploads (auto-cleaned)
```

### Server API Overview

| Route | Method | Description |
|-------|--------|------------|
| `/api/pdf/compress` | POST | Compresses a PDF using server-side optimization |
| `/api/pdf/convert` | POST | Converts PDFs to other formats server-side |
| `/api/image/resize` | POST | Resizes images using `sharp` (high quality) |
| `/api/image/compress` | POST | Compresses images server-side with `sharp` |
| `/api/engineer/*` | POST | Developer tool processing (minification, etc.) |
| `/api/researcher/*` | POST | Research tool processing (data analysis, etc.) |

## 🔄 Data Flow Diagram

```
User selects a file
        │
        ▼
   Tool Runner (.jsx)
        │
        ├──► Client-Side Processing (browser)
        │         │
        │         ▼
        │    src/services/{pdf,image}/
        │         │ uses pdf-lib, pdfjs-dist,
        │         │      Canvas API, docx, xlsx
        │         ▼
        │    Processed File → Download
        │
        └──► Server-Side Processing (heavy tasks)
                  │
                  ▼
            serverProcessingService.js
                  │ HTTP POST with FormData
                  ▼
            server/routes/{pdf,image}.js
                  │ uses sharp, pdf-lib
                  ▼
            Processed File → Response → Download
```

## 🔐 Authentication Flow

```
User visits protected route
        │
        ▼
  AuthContext checks Firebase auth state
        │
  ├── Logged in ──► Render page
  └── Not logged in ──► Redirect to /auth
                              │
                         AuthPage.jsx
                              │
                   ┌──────────┴──────────┐
                   │                     │
              Email/Password       Google Sign-In
                   │                     │
                   └────────┬────────────┘
                            │ Firebase Auth
                            ▼
                    AuthContext updates
                            │
                            ▼
                    Redirect to /profile
